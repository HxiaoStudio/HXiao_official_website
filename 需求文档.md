### 1.1 品牌背景

**「浮晓 HXiao」** 是由创作者基于共同价值观自愿组成的互联网非营利性创作共同体，对外统一使用「浮晓 HXiao」作为品牌标识。

#### 品牌标识（Logo）

工作室采用 **「HXiao Studio」** 双层文字组合作为品牌核心标识：主层 **HXiao**（喷溅毛边手写体）为视觉焦点，辅层 **Studio**（无衬线规整小字）平衡画面，象征「热爱的温度」与「制度的克制」共存。配色为纯黑 `#000000` + 纯白 `#FFFFFF` 的极简双色调，作为整站粉彩色系的「冷静锚点」。

* **官方文件**：`HXiao_logo.jpg`（位于项目根目录 `assets/brand/HXiao_logo.jpg`）

> 完整的 Logo 使用规范（构成、安全区、最小尺寸、配色版本、禁用场景、衍生应用等）见 **四、视觉设计需求 - 4.5 Logo 与品牌标识规范**。

* **B站主页**：[space.bilibili.com/1232406878](https://space.bilibili.com/1232406878)
* **微信公众号**：HXiao\_Studio
* **代表作品**：[BV1FzsNzbE9g](https://www.bilibili.com/video/BV1FzsNzbE9g)（播放量最高视频）

### 1.2 核心理念

> **生如蜉蝣，却执意丈量天光的刻度；**
> **仰望星海，亦于深渊回响中听见自己的心跳。**

**名字释义：**

* **浮** — 取蜉蝣之浮，记录微小日常的脉搏
* **晓** — 取知晓之晓，探寻广阔世界的模样

**精神内核：** 渺小，但绝不停止思考与远望
**组织承诺：** 长期非营利、纯公益，只为热爱与共鸣而生

### 1.3 组织制度

工作室实行**成员民主协作制**，所有成员地位平等，共同决策。设有《浮游亦晓工作室章（临时）》规范内部事务，待成员构成完整后咨询社员意见进行修订。

***

**第一章 总则**

**第一条 名称与性质**

1. 本组织全称为"浮游亦晓工作室"，英文名称为"HXiaoStudio"，对外统一使用"浮晓 HXiao"作为品牌标识。
2. 本工作室是由创作者基于共同价值观自愿组成的互联网非营利性创作共同体，实行自主管理、民主协作。

**第二条 宗旨**

1. 遵守国家法律法规，践行社会主义核心价值观，传播积极健康的网络文化。
2. 以"渺小，但绝不停止思考与远望"为精神内核，通过创作记录真实、探索世界、启发思考。
3. 坚持非营利与公益导向，所有活动基于热爱与共鸣，为社会贡献温暖力量。

**第三条 组织原则**

1. 实行成员民主协作制。所有成员地位平等，共同决策，但角色与责任不同。
2. 工作室最高议事机构为全体成员会议，日常协调与决策机构为核心成员小组。
3. 决策一般遵循"全体成员过半同意"生效原则，章程修改等重大事项需三分之二以上同意。

**第二章 成员定义与分类**

**第四条 成员总则**

1. 本工作室成员均为自愿加入、认同章程并参与内容共创的个体。
2. 外部投稿者不属于成员，其与工作室仅为基于《投稿授权协议》的单次或有限次内容合作，不享有成员权利，亦不承担成员义务。

**第五条 成员分类**
工作室成员分为以下两类：

1. 核心成员
   * 指长期深度参与工作室创作、运营或管理，并进入核心成员小组的成员。
   * 包括：负责人、各职能方向的固定召集人或骨干成员。
   * 义务：承担更多组织、协调、决策与日常维护责任，是工作室稳定发展的基石。
   * 权利：参与核心成员小组议事，对日常事务拥有提案权与表决权。
2. 普通成员
   * 指认同工作室价值观，参与项目共创，但未进入核心成员小组的成员。
   * 义务：遵守章程，参与所报名的项目协作，尊重团队决策。
   * 权利：参与全体成员会议，对重大事务拥有提案权与表决权；可自由参与项目，享有创作署名权。

**第六条 成员身份转换**

1. 普通成员可通过持续贡献、主动承担更多责任，经核心成员小组提名并获全体成员会议简单多数同意，成为核心成员。
2. 核心成员若因个人原因希望减轻责任，可申请转为普通成员，其在小组的职务由小组另行安排。

**第三章 组织架构与职责**

**第七条 组织架构**
工作室实行扁平化管理，设立三级架构：

* 全体成员会议：由全体核心成员与普通成员组成。
* 核心成员小组：由负责人及各职能方向的核心成员组成。
* 项目小组：为特定内容项目临时组建，实行"项目负责人制"。

**第八条 全体成员会议职责**

1. 修改本章程。
2. 选举与罢免负责人。
3. 审议年度计划、财务报告等重大事项。
4. 决定核心成员的晋升与除名。

**第九条 负责人选举与职责**

1. 负责人由全体成员会议选举产生，任期一年，可连任一届。
2. 负责人是工作室的对外代表与总协调人，主持核心成员小组会议，落实各项决议，无独断权。
3. 负责人可由全体成员会议罢免（需三分之二同意）。

**第十条 核心成员小组职责**

1. 审核并决议新成员的加入申请。
2. 负责工作室日常运营、资源协调与项目进度跟踪。
3. 审议普通项目提案，制定执行计划。
4. 组织成员培训与协作规范维护。
5. 会议决议需记录并向全体成员公开摘要。

**第四章 成员加入、退出与除名**

**第十一条 加入程序**

1. 申请人须认同本章程及工作室价值观，通过指定渠道提交申请。
2. 申请由核心成员小组进行审核与评估。
3. 经核心成员小组过半数成员同意，即可批准加入。
4. 批准后，申请人完成电子确认流程，正式成为工作室普通成员。
5. 新成员由核心成员小组指定一名现有成员协助熟悉工作规范。

**第十二条 退出机制**

1. 成员可自愿退出，但应提前告知并完成工作交接。
2. 核心成员退出时，应推荐接替人选或完成职责移交。

**第十三条 除名机制**
成员出现以下情况，经核心成员小组提议，全体成员会议三分之二以上同意，可予以除名：

1. 严重违反国家法律法规或本章程；
2. 蓄意损害工作室声誉或利益；
3. 长期不履行成员基本义务且无合理解释；
4. 其他经集体认定严重不当的行为。

**第五章 创作、版权与署名规范**

**第十四条 内容创作原则**

1. 坚持原创优先，鼓励独特视角与真诚表达。
2. 恪守版权法律，严禁使用任何未经合法授权、明确禁止二次创作或超出授权范围的素材。

**第十五条 版权归属原则**
遵循《共创伙伴加入确认书》与《投稿授权协议》约定的分层版权制度，明确区分个人素材版权、成品发布权与完整作品版权。

**第十六条 最终成品署名规范（强制性）**
为尊重所有贡献者并履行版权义务，所有发布成品须包含清晰署名：

1. 署名位置：视频须在片尾、图文须在文末/描述区明确列出。
2. 署名标题：统一使用"【浮晓 HXiao】创作团队"。
3. 署名内容与顺序：
   * 核心创作人员：按策划、文案、拍摄、剪辑、设计、配音等职能列出参与者昵称。
   * 外部投稿者：注明投稿人昵称及"投稿作者"身份。
   * 素材来源声明：详细列出所有非原创素材信息，格式示例：
     * 音乐：《曲名》- 作者/演出者（来源：HOYOMiX；授权类型：CC-BY-NC 3.0）
     * 影像/图片：《素材名》- 作者（来源：Pexels；授权：CC0）
     * 字体：字体名称 - 版权方（授权类型）
   * 特别鸣谢：对提供重要支持者予以感谢。
4. 原创性与合规承诺：项目负责人须在发布前核对署名完整性与素材授权合规性，并对该环节负最终责任。

**第十七条 违规处理**

### 5.5 数据获取与 API 集成

**B站视频数据采用客户端直接请求方案（方案一）**，以保证数据的实时性，同时避免静态站点需要后端服务的复杂性。

#### 5.5.1 接口说明

| 用途 | 接口 URL | 备注 |
|---|---|---|
| 用户视频列表 | `https://api.bilibili.com/x/space/wbi/arc/search?mid=1232406878&ps=30&pn=1` | `mid` 为B站UID，`ps` 为每页数量，`pn` 为页码 |
| 单个视频信息 | `https://api.bilibili.com/x/web-interface/view?bvid=...` | 可获取详细播放量、点赞数等 |
| 用户信息 | `https://api.bilibili.com/x/space/wbi/acc/info?mid=1232406878` | 获取粉丝数、总播放量等 |

* 接口无需 API Key，跨域支持良好（CORS 通常为 `*`）。
* 部分接口（如 `wbi` 路径）需附带 `wbi` 签名参数，签名算法公开，可在前端 JavaScript 中实现。
* 若希望避免签名处理，可暂时使用旧版接口（如 `https://api.bilibili.com/x/space/arc/search?mid=...`），但新版更稳定，推荐使用并实现签名。

#### 5.5.2 前端实现方式

* 在页面加载时，通过 `fetch` 调用上述接口。
* 解析返回的 JSON，提取视频标题、封面、播放量、点赞数、发布时间等字段，动态渲染至作品页面。
* 对于总播放量、粉丝数等汇总数据，可从用户信息接口获取，并配合数字滚动动效展示。
* 为了减少重复请求，可在客户端利用 `localStorage` 设置缓存，缓存有效期建议为 30 分钟至 1 小时（视频数据非实时变化可容忍一定延迟）。

**示例代码结构**（示意）：

```javascript
async function fetchVideoList(uid) {
  const cacheKey = `bili_videos_${uid}`;
  const cached = localStorage.getItem(cacheKey);
  if (cached) {
    const { data, timestamp } = JSON.parse(cached);
    if (Date.now() - timestamp < 1800000) return data; // 30分钟缓存
  }
  
  const response = await fetch(`https://api.bilibili.com/x/space/wbi/arc/search?mid=${uid}&ps=30`);
  const json = await response.json();
  if (json.code === 0) {
    const videos = json.data.list.vlist;
    localStorage.setItem(cacheKey, JSON.stringify({ data: videos, timestamp: Date.now() }));
    return videos;
  }
  throw new Error('Failed to fetch');
}
```

#### 5.5.3 注意事项

1. **合规性**：B站未禁止此类个人网站调用，但需合理控制频率。客户端缓存可避免短时间内高频请求。
2. **错误处理**：需处理网络异常、接口变动等场景，可显示静态占位信息或上一次缓存数据。
3. **数据展示**：播放量等数值可能较大，前端需格式化（如超过 1 万显示为 `1.2万`）。
4. **跨域与签名**：若遇到跨域限制（极少情况），或必需 `wbi` 签名，可引入社区维护的轻量签名库，或回退至构建时预取方案（方案二）。
