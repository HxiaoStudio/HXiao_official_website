# 浮晓 HXiao 官网 · 开发规范

---

## 1 项目概览

### 1.1 技术栈

| 维度 | 选型 | 说明 |
|---|---|---|
| 框架 | **Astro 6**（`output: 'static'`） | 文件路由，每个 `.astro` 编译成一个静态 HTML 页面 |
| 样式 | **Tailwind CSS 3** + CSS 变量 | utility-first，主题色用 CSS 变量 + Tailwind 映射 |
| 语言 | TypeScript（`strict` 模式） | `tsconfig.json` 继承 `astro/tsconfigs/strict` |
| 内容 | Markdown + JSON 数据驱动 | 章程等长文走 MD，列表数据走 JSON |
| SEO | `@astrojs/sitemap` | 构建时自动生成 `sitemap-index.xml` |
| 部署 | GitHub Pages | push 到 `main` 触发 CI 自动构建发布 |

### 1.2 目录结构

```
HXiao_html/
├── src/                      ← 源码（所有开发都在这里）
│   ├── pages/                ← 页面层：文件路由，1 个 .astro = 1 个 URL
│   ├── layouts/              ← 布局层：页面外壳（BaseLayout）
│   ├── components/           ← 组件层：可复用 UI 块
│   ├── data/                 ← 数据层：JSON / MD 内容源
│   ├── styles/               ← 样式层：global.css（单一全局样式表）
│   ├── scripts/              ← 脚本层：可复用前端工具（如 pdf-guard.js）
│   └── utils/                ← 工具层：构建期/服务端工具（如 bilibili.ts）
├── public/                   ← 直接拷贝到产物根的静态资源（CNAME / robots / assets / fonts）
├── 文档/                      ← 业务/法务/工程文档（本文件所在）
├── dist/                     ← 构建产物（gitignore，勿手动改）
├── astro.config.mjs          ← Astro 配置
├── tailwind.config.mjs       ← Tailwind 配置（主题色映射）
└── package.json
```

开发只在 `src/` 内进行。`public/` 只放"原样拷贝"的资源（图片、字体、CNAME、robots）。

### 1.3 构建与部署

```
本地：npm run dev      → Astro 开发服务器（热更新）
       npm run build   → 输出到 dist/
       npm run preview → 本地预览构建产物

部署：git push origin main
       → GitHub Actions (.github/workflows/deploy.yml)
       → npm ci → npm run build → 发布到 GitHub Pages
       → 生效域名 hxiaostudio.cn（public/CNAME 声明）
```

## 2 分层架构

### 2.1 六层分层模型

本项目按以下六层组织，依赖方向**只能自上而下**：

```
┌─────────────────────────────────────────────────────────┐
│  Layout 层   layouts/BaseLayout.astro                    │
│  职责：页面外壳（<html>/<head>/<body>骨架、Navbar、Footer、 │
│        SEO meta、全局动效观察器）。所有页面必须套这层。     │
└────────────────────────┬──────────────────────────────── ┘
                         │ 套用（<BaseLayout> 包裹）
┌────────────────────────▼──────────────────────────────── ┐
│  Page 层     pages/*.astro                                │
│  职责：一个页面 = 一个文件 = 一个 URL。负责页面级内容编排、 │
│        组装组件、注入页面专属数据。不重复写 head/body。     │
└────────────────────────┬──────────────────────────────── ┘
                         │ 引用、组合
┌────────────────────────▼──────────────────────────────── ┐
│  Component 层 components/*.astro                          │
│  职责：可复用 UI 块（Navbar / Footer / Modal / 卡片等）。   │
│        通过 props 接收数据，不直接读全局状态。              │
└────────────────────────┬──────────────────────────────── ┘
                         │ 引用数据 / 样式 / 脚本
┌────────────────────────▼──────────────────────────────── ┐
│  Data 层     data/*.json | data/*.md                      │
│  Style 层    styles/global.css                            │
│  Script 层   scripts/*.js | utils/*.ts                    │
│  ── 这三层是"叶子层"：只被上层引用，不反向依赖任何上层 ──  │
└─────────────────────────────────────────────────────────┘
```

上层可以引用下层，下层**绝不**反向引用上层。Data/Style/Script 是叶子层，不能反向引用 Component 或 Page。

### 2.2 各层职责清单

| 层 | 放什么 | 不放什么 |
|---|---|---|
| **Layout** | `<html>`/`<head>`/`<body>` 骨架、全局 meta、Navbar、Footer、全局脚本（如淡入观察器） | 页面专属内容、业务逻辑 |
| **Page** | 该 URL 的内容编排、页面专属数据获取、组装组件 | 重复的 head/body 结构、可被多页复用却不抽成组件的 UI |
| **Component** | 可复用的 UI 单元，通过 props 接收数据 | 硬编码的业务数据、页面专属逻辑 |
| **Data** | 易变的列表/内容数据（JSON）、长文（MD） | 代码逻辑、组件渲染 |
| **Style** | 设计 token、全局语义类、Tailwind 指令 | 散落的内联样式、重复的选择器 |
| **Script** | 跨页面复用的前端工具函数、构建期/服务端工具 | 只服务单个页面的脚本（应留在该页面 `<script>`） |

### 2.3 新增页面决策流

```
要新增一个页面 / URL
        │
        ▼
┌───────────────────────────┐
│ 是否需要主导航露出？       │── 否 ──┐
└──────────┬────────────────┘        │
           │ 是                       │
           ▼                          │
  改 Navbar + Footer 的导航数据       │
  （两处都改，或走 3.1 单一数据源       │
│   抽 data/nav.ts）                  │
           │                          │
           ▼                          │
┌───────────────────────────┐         │
│ 是否是隐藏页（noindex）？  │         │
└──────────┬────────────────┘         │
           │                          │
     ┌─────┴─────┐                    │
     ▼           ▼                    ▼
   是隐藏页    是正常页            不进导航
     │           │                    │
     ▼           ▼                    ▼
  仍用 BaseLayout（传 noindex prop）   仍用 BaseLayout
     │           │                    │
     └─────┬─────┘────────────────────┘
           ▼
   创建 src/pages/<路由>.astro
           │
           ▼
   在页面里只写"内容编排"，套 <BaseLayout title=...>
           │
           ▼
   若有可复用 UI → 抽 Component
   若有页面专属逻辑 → 留在本页 <script>
   若有跨页逻辑 → 抽 src/scripts/
```

任何页面（含隐藏页、流程内部页）都必须套 `BaseLayout`。若 BaseLayout 不支持所需功能（如隐藏导航），**扩展 BaseLayout 的 props**，禁止绕开它手写 `<html><head><body>`。

## 3 编码原则

### 3.1 单一数据源

任何一份"会被多处引用"的数据，**只在一处定义**。抽到 `src/data/` 下的数据文件或 `src/config.ts`，所有引用方 import 同一份。

### 3.2 Props 下传，不跨层、不全局

组件需要数据，由父级通过 props 传入；**禁止**组件内部直接 `import` 业务页面、**禁止**读写全局变量传递状态。

例外：浏览器原生 API（`localStorage`、`document`）和构建期数据（`Astro.glob`、`import json`）不算"全局状态"。

### 3.3 第三次重复即抽象（Rule of Three）

同一段逻辑或结构：
- **第 1 次**：直接写。
- **第 2 次**：可以复制，但要在原处留 `// TODO: 与 xxx 重复，考虑抽象` 注释。
- **第 3 次**：**必须**抽成公共组件 / 工具函数 / 数据源。

### 3.4 不写无效修改

- 禁止仅换写法但不改变行为的修改。
- 禁止为了"看起来更规范"而大规模重命名、移动文件，除非有实质性改进并经过评估。
- 可以清理死代码和已注释代码。

### 3.5 不让简单逻辑变复杂

- 一段逻辑能用 10 行清楚表达，就不该被拆成 3 个抽象层、2 个工具函数、1 个配置文件。
- 嵌套深度 > 3 层是危险信号，出现时应考虑能否改写得更扁平。

### 3.6 不过度防御

只在"可能真实发生"的边界做防御：

- ✅ 用户输入 → 校验
- ✅ 外部 API 返回值 → 校验
- ✅ 文件读取 / 路径拼接 → 校验
- ❌ Astro 框架保证的 `Astro.props` → 不要加 `if (!props) return null`
- ❌ 构建期已知存在的 import JSON → 不要加 `try-catch` 兜底
- ❌ 只在单个页面使用的脚本 → 不要加"国际化预备""多主题兼容"等未来才可能需要的分支

## 4 命名约定

### 4.1 文件命名

| 类型 | 规则 | 示例 |
|---|---|---|
| 页面 `src/pages/` | 全小写 **kebab-case**，嵌套路由用目录 | `index.astro`、`about.astro`、`join/confirm.astro` |
| 组件 `src/components/` | **PascalCase** | `Navbar.astro`、`WxQrcodeModal.astro` |
| 布局 `src/layouts/` | **PascalCase** | `BaseLayout.astro` |
| 工具/脚本 `src/scripts/` `src/utils/` | **kebab-case**（多词用连字符） | `pdf-guard.js`、`bilibili.ts` |
| 数据 `src/data/` | **kebab-case**，按业务域命名 | `members.json`、`articles.json`、`charter.md` |
| 静态资源 `public/assets/` | **kebab-case**，按用途分子目录 | `assets/brand/HXiao_logo.png`、`assets/qrcode/` |

### 4.2 CSS 类命名

- **业务语义类**（写入 `global.css`）：扁平 **kebab-case**，无 BEM 的 `__` / `--`。
  - ✅ `.fade-in-up`、`.card-hover`、`.nav-glass`、`.container-main`、`.print-header`
  - ❌ `.fadeInUp`（驼峰）、`.nav__link--active`（BEM）、`.btn_glow`（下划线）
- **Tailwind utility**：直接用 Tailwind 原生类，不在 `global.css` 重复定义。
- **主题色类**：通过 `tailwind.config.mjs` 映射为语义化类名——`bg-bg-main`、`text-text-primary`、`border-border`、`text-accent`。
- **类名一律英文**，禁止拼音、禁止中英混合。

### 4.3 JS / TS 命名

- 变量、函数：**camelCase** —— `fetchVideoMeta`、`shouldBlockPdfOnMobile`、`formatChineseDate`
- 接口、类型：**PascalCase** —— `VideoMeta`、`Member`、`ProjectCard`
- 常量（真正不可变的配置）：**UPPER_SNAKE_CASE** —— `WX_ID`
- 布尔变量/props：用 `is`/`has`/`should` 前缀 —— `isDark`、`hasNav`、`shouldBlock`

### 4.4 数据文件键名

JSON 键统一 **camelCase**，与 JS 一致。

## 5 代码风格

### 5.1 缩进与换行

- **统一 2 空格缩进**，禁止 Tab。
- 行尾 **LF**，文件末尾保留一个空行，去除行尾空格。
- 由 `.editorconfig` + `.prettierrc` 强制。

### 5.2 HTML / Astro 标签风格

- 属性值用**双引号**：`class="..."`、`href="..."`。
- 空元素自闭合 XHTML 风格：`<meta charset="UTF-8" />`、`<img ... />`、`<link ... />`。
- 属性顺序（非强制，建议）：`class` → `id` → `type/href/src` → `aria-*` → 其它。
- 长属性换行时，每行一个属性、对齐缩进。

### 5.3 注释风格

| 场景 | 风格 | 示例 |
|---|---|---|
| `.astro` 文件首行 | `// 文件名.astro - 中文职能描述` | `// BaseLayout.astro - 全局布局组件` |
| 行内中文注释 | `// 中文说明` | `// 视差效果` |
| TS/JS 工具函数 | **JSDoc 块注释**（`/** ... */`） | 见 `pdf-guard.js`、`bilibili.ts` |
| CSS 分节 | `/* ===== 标题 ===== */` | `/* ===== 动效相关 ===== */` |

每个 `.astro` 文件首行必须有职能注释，无一例外。

### 5.4 中英文排版（盘古之白）

- **中文与英文/数字之间加一个空格**：`通过 BVID 从 B站 API 获取视频元数据`、`PC 端`、`GitHub Pages`。
- 标点符号：正文用中文全角（，。：；？！），代码、URL、技术标识用半角。
- 品牌名统一：`浮晓 HXiao` / `浮晓 HXiao Studio`。

## 6 数据与样式

### 6.1 内容放置决策表

| 内容类型 | 放哪里 | 示例 |
|---|---|---|
| **易变的列表数据**（会增删条目） | `src/data/*.json` | `members.json`、`updates.json`、`videos.json`、`articles.json` |
| **长文 / 结构化正文** | `src/data/*.md`，用 Astro 渲染 | `charter.md`（章程） |
| **页面专属静态文案**（只这一页用、很少改） | 直接写在 `.astro` 里 | 各页标题、段落正文 |
| **跨页面复用的常量**（邮箱、链接、ID） | **`src/config.ts`** | 邮箱、B站空间、公众号 ID |

新增"会持续更新"的内容（如成员、动态、文章列表），**优先**走 `src/data/*.json` + 页面 `.map()` 渲染。

### 6.2 站点常量必须集中

所有"会被 2 处以上引用"的站点常量，**禁止**在 `.astro` 里硬编码，必须集中到 `src/config.ts`。例如：邮箱、B站空间 URL、公众号 ID。

### 6.3 设计 token 体系

主题色通过 **CSS 自定义属性（变量）** 定义，再由 `tailwind.config.mjs` 映射为 Tailwind 类。**禁止**在 `.astro` 里硬编码 `#AEDBFC` 这类颜色值。

```
global.css :root { --color-accent: 174 219 252; ... }
                ↓ tailwind.config.mjs 映射
                ↓
页面里用：text-accent / bg-accent / border-accent
```

完整的 token 列表见 `src/styles/global.css` 开头，涵盖：背景色、文字色、强调色、辅助色、品牌色，并为深色模式（`.dark`）定义了对应的值。

### 6.4 utility 与语义类选择

| 情况 | 做法 |
|---|---|
| 一次性、页面专属的布局/间距 | 直接 Tailwind utility，写在 `class="..."` |
| 跨多处复用的视觉模式（按钮样式、卡片悬停、淡入动效） | 写成语义类进 `global.css` |
| 带复杂选择器 / 伪元素 / `@media` 的样式 | 写进 `global.css`（utility 表达不了） |
| 组件专属、不想污染全局的样式 | 用 Astro **scoped `<style>`**（如 `charter.astro`、`ThirdPartyModal.astro`） |

### 6.5 打印样式规范

多个打印区必须共用一个公共类（如 `.print-doc`）加到容器上，CSS 只写一份 `.print-doc .print-header { ... }`。

**新增打印页时**：必须用 `.print-doc` 公共类，禁止再复制一份针对新 ID 的样式。

### 6.6 `!important` 使用约束

仅允许在两类"强制覆盖"场景使用：
- `@media print`（强制浅色、强制隐藏非打印区）
- `@media (prefers-reduced-motion: reduce)`（强制关闭动画）

其它场景**禁止**用 `!important` 解决优先级问题——应通过调整选择器层级解决。

## 7 可访问性与 SEO

### 7.1 必须项清单

- [ ] `<html lang="zh-CN">`（由 `BaseLayout` 保证，**不要**在自己写的页面里覆盖）。
- [ ] 页面有 `title`（通过 `<BaseLayout title="...">` 传入）。
- [ ] 有 `meta description`（`BaseLayout` 有默认值，重要页面应通过 prop 覆盖，参考 `projects.astro`）。
- [ ] Open Graph + Twitter Card meta（由 `BaseLayout` 注入，无需手写）。
- [ ] 所有 `<img>` 必有 `alt`，**禁止空 alt**。
- [ ] 所有图标型 `<button>` 必有 `aria-label`（如主题切换、汉堡菜单、社交链接）。
- [ ] 弹窗用 `role="dialog"` + `aria-modal="true"` + `aria-labelledby`，并支持 ESC 关闭（参考 `WxQrcodeModal.astro`、`ThirdPartyModal.astro`）。
- [ ] 动效必须响应 `@media (prefers-reduced-motion: reduce)`（`global.css` 已有全局降级，JS 动画也要判断，参考 `StarCanvas.astro`）。
- [ ] 隐藏页（如流程内部页）加 `<meta name="robots" content="noindex, nofollow">`。

### 7.2 Logo `alt` 统一

同一张 Logo 的 `alt` 统一为 `浮晓 HXiao Studio`（与正文品牌标识一致）。

### 7.3 语义化标签

优先使用语义化标签：`<header>`、`<nav>`、`<main>`、`<article>`、`<section>`、`<aside>`、`<footer>`、`<time>`、`<details>`/`<summary>`（参考 `charter.astro` 的 `<article>`、`join.astro` 的 `<details>` FAQ）。

## 8 Git 协作

### 8.1 提交信息前缀规范

每条提交信息必须以中文类型前缀开头，格式：`[类型] 简述`。

| 前缀 | 用于 | 示例 |
|---|---|---|
| `[新功能]` | 新增功能、新页面、新组件 | `[新功能] 集成站点地图 Sitemap 自动生成` |
| `[修复]` | 修复 bug、修复错误 | `[修复] contact 页面 GitHub 链接地址拼写错误` |
| `[优化]` | 性能、体验、可访问性、SEO 改进（不改变功能） | `[优化] 深色模式切换动画` |
| `[重构]` | 改变代码结构、不改变对外行为 | `[重构] projects 页面改造为项目聚合入口` |
| `[样式]` | 纯样式调整（CSS、Tailwind 类、视觉细节） | `[样式] 调整首页卡片间距` |
| `[内容]` | 文案、数据内容更新（不改代码逻辑） | `[内容] 更新成员列表` |
| `[配置]` | 配置文件、依赖版本、构建配置 | `[配置] 升级 Astro 至 6.4.8` |
| `[部署]` | CI、部署、域名相关 | `[部署] 配置 GitHub Actions 自动发布` |
| `[文档]` | 文档变更（含本规范文档） | `[文档] 新增开发规范` |

类型与内容匹配，简述用祈使句或陈述句，中英文间加空格，提交前检查括号闭合。

### 8.2 提交粒度

- **一个提交做一件事**：不要把"加新功能 + 改无关 bug + 改样式"塞进一个提交。
- 遇到重构时，**每个治理点独立一个提交**（便于回滚和审查）。

### 8.3 分支与发布

- `main` 分支即生产分支，push `main` 自动触发部署。
- 较大的改动建议开分支开发，合并到 `main` 再部署。

## 9 工程化工具链

| 工具 | 文件 | 作用 |
|---|---|---|
| **EditorConfig** | `.editorconfig` | 统一缩进（2 空格）、换行（LF）、字符集（UTF-8）、去行尾空格。编辑器原生支持，无需装包。**对 `.astro` 文件同样生效**，是当前 `.astro` 格式的主要约束。 |
| **Prettier** | `.prettierrc` + `.prettierignore` | 格式化约束。已装 `prettier-plugin-astro` 备用，但**当前 `.astro` 文件被 `.prettierignore` 暂时排除**，Prettier 实际只格式化 `.ts/.js/.css/.json/.mjs` 等非 Astro 文件。 |
| **npm scripts** | `package.json` | `npm run format`（一键格式化）、`npm run format:check`（检查不改动）。 |

**使用方式**：
- 写完代码执行 `npm run format` 自动格式化。
- 想检查代码是否符合规范（不改文件）执行 `npm run format:check`。
- 大批量历史文件的格式化（如缩进修复）建议单独一个 `[重构]` 提交。
