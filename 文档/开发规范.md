# 浮晓 HXiao 官网 · 开发规范

## §1 项目概览

### 1.1 技术栈

> **本项目是 Astro + Tailwind 的静态站点生成（SSG）项目，不是"纯 HTML/CSS/JS"。**

| 维度 | 选型 | 说明 |
|---|---|---|
| 框架 | **Astro 6**（`output: 'static'`） | 文件路由，每个 `.astro` 编译成一个静态 HTML 页面 |
| 样式 | **Tailwind CSS 3** + CSS 变量 | utility-first，主题色用 CSS 变量 + Tailwind 映射 |
| 语言 | TypeScript（`strict` 模式） | `tsconfig.json` 继承 `astro/tsconfigs/strict` |
| 内容 | Markdown + JSON 数据驱动 | 章程等长文走 MD，列表数据走 JSON |
| SEO | `@astrojs/sitemap` | 构建时自动生成 `sitemap-index.xml` |
| 部署 | GitHub Pages | push 到 `main` 触发 CI 自动构建发布 |

⚠️ **`文档/需求文档.md` 中"静态站点（HTML/CSS/JS）"的描述已过时**。

### 1.2 目录结构

```
HXiao_html/
├── src/                      ← 源码（所有开发都在这里）
│   ├── pages/                ← 页面层：文件路由，1 个 .astro = 1 个 URL
│   ├── layouts/              ← 布局层：页面外壳（BaseLayout）
│   ├── components/           ← 组件层：可复用 UI 块
│   ├── data/                 ← 数据层：JSON / MD 内容源
│   ├── styles/               ← 样式层：global.css（单一全局样式表）
│   ├── scripts/              ← 脚本层：可复用前端工具（如 pdf-guard.js）
│   └── utils/                ← 工具层：构建期/服务端工具（如 bilibili.ts）
├── public/                   ← 直接拷贝到产物根的静态资源（CNAME / robots / assets / fonts）
├── 文档/                      ← 业务/法务/工程文档（本文件所在）
├── dist/                     ← 构建产物（gitignore，勿手动改）
├── astro.config.mjs          ← Astro 配置
├── tailwind.config.mjs       ← Tailwind 配置（主题色映射）
└── package.json
```

**关键原则**：开发只在 `src/` 内进行。`dist/` 是构建产物，`public/` 只放"原样拷贝"的资源（图片、字体、CNAME、robots）。

### 1.3 构建与部署流程

```
本地：npm run dev      → Astro 开发服务器（热更新）
       npm run build   → 输出到 dist/
       npm run preview → 本地预览构建产物

部署：git push origin main
       → GitHub Actions (.github/workflows/deploy.yml)
       → npm ci → npm run build → 发布到 GitHub Pages
       → 生效域名 hxiaostudio.cn（public/CNAME 声明）
```

**不要**手动改 `dist/`、不要在 CI 流程外发布、不要绕过 `main` 分支部署。

## §2 目标架构（核心解耦设计）

> 这一章是本文档的重心。理解了它，后面所有章节都是它的细化。

### 2.1 六层分层模型

本项目按以下六层组织，依赖**只能自上而下**：

```
┌─────────────────────────────────────────────────────────┐
│  Layout 层   layouts/BaseLayout.astro                    │
│  职责：页面外壳（<html>/<head>/<body>骨架、Navbar、Footer、 │
│        SEO meta、全局动效观察器）。所有页面必须套这层。     │
└────────────────────────┬──────────────────────────────── ┘
                         │ 套用（<BaseLayout> 包裹）
┌────────────────────────▼──────────────────────────────── ┐
│  Page 层     pages/*.astro                                │
│  职责：一个页面 = 一个文件 = 一个 URL。负责页面级内容编排、 │
│        组装组件、注入页面专属数据。不重复写 head/body。     │
└────────────────────────┬──────────────────────────────── ┘
                         │ 引用、组合
┌────────────────────────▼──────────────────────────────── ┐
│  Component 层 components/*.astro                          │
│  职责：可复用 UI 块（Navbar / Footer / Modal / 卡片等）。   │
│        通过 props 接收数据，不直接读全局状态。              │
└────────────────────────┬──────────────────────────────── ┘
                         │ 引用数据 / 样式 / 脚本
┌────────────────────────▼──────────────────────────────── ┐
│  Data 层     data/*.json | data/*.md                      │
│  Style 层    styles/global.css                            │
│  Script 层   scripts/*.js | utils/*.ts                    │
│  ── 这三层是"叶子层"：只被上层引用，不反向依赖任何上层 ──  │
└─────────────────────────────────────────────────────────┘
```

**依赖方向铁律**：上层可以引用下层，下层**绝不**反向引用上层。
- ✅ `Page` 引用 `Component`、`Data`、`Style`、`Script`
- ✅ `Component` 引用 `Data`、`Style`、`Script`
- ❌ `Data`/`Style`/`Script` 引用 `Component` 或 `Page`（叶子层不可回头）

### 2.2 各层职责清单

| 层 | 放什么 | 不放什么 |
|---|---|---|
| **Layout** | `<html>`/`<head>`/`<body>` 骨架、全局 meta、Navbar、Footer、全局脚本（如淡入观察器） | 页面专属内容、业务逻辑 |
| **Page** | 该 URL 的内容编排、页面专属数据获取、组装组件 | 重复的 head/body 结构、可被多页复用却不抽成组件的 UI |
| **Component** | 可复用的 UI 单元，通过 props 接收数据 | 硬编码的业务数据、页面专属逻辑 |
| **Data** | 易变的列表/内容数据（JSON）、长文（MD） | 代码逻辑、组件渲染 |
| **Style** | 设计 token、全局语义类、Tailwind 指令 | 散落的内联样式、重复的选择器 |
| **Script** | 跨页面复用的前端工具函数、构建期/服务端工具 | 只服务单个页面的脚本（应留在该页面 `<script>`） |

### 2.3 三条解耦原则

**原则一：单一数据源（Single Source of Truth）**

任何一份"会被多处引用"的数据，**只在一处定义**。

典型反面案例（当前项目待治理）：
- 导航项 `navItems` 在 `Navbar.astro` 和 `Footer.astro` 各硬编码一份（9 项完全相同）。
- 邮箱 `hxiao_studio@163.com` 散落在 8+ 处（commit `c0285bb` 因这种散落导致 GitHub 链接漏改拼写错误）。

治理方向：抽到 `src/data/` 下的数据文件或 `src/config.ts`，所有引用方 import 同一份。

**原则二：Props 下传，不跨层、不全局**

组件需要数据，由父级通过 props 传入；**禁止**组件内部直接 `import` 业务页面、`禁止`读写全局变量传递状态。

例外：浏览器原生 API（`localStorage`、`document`）和构建期数据（`Astro.glob`、`import json`）不算"全局状态"。

**原则三：第三次重复即抽象（Rule of Three）**

> 同一段逻辑或结构：
> - **第 1 次**：直接写。
> - **第 2 次**：可以复制，但要在原处留 `// TODO: 与 xxx 重复，考虑抽象` 注释。
> - **第 3 次**：**必须**抽成公共组件 / 工具函数 / 数据源，否则就是在制造屎山。

当前项目最典型的违反：`submit.astro` / `join.astro` / `join/confirm.astro` 三份"取表单值 → 填打印区 → `window.print()`"脚本近乎逐行重复。

### 2.4 代码极简原则（反过度设计）

AI Coding 的另一面是"过度积极"——为了稳妥而加不必要的抽象和防御。以下四条底线防止代码走向另一个极端。

**① 不提前抽象（YAGNI）**

"第 1 次直接写，第 2 次留注释，第 3 次才抽象"（§2.3 原则三）已经隐含了这条，但值得单独强调：**禁止在第 1 次使用时预判"以后可能会复用"而提前抽组件/函数。** 等到真正出现第 2 次、第 3 次时再抽象，那时需求是明确的，抽象形状是准确的。

反例：为某个页面写了一个按钮样式，立即抽成 `<Button variant="...">` 组件——但该按钮样式只在这一处用到，其他页面没有类似需求。这是过度设计。

**② 不写无效修改**

- 禁止单纯的"换一种写法但行为不变"的变更（如把一个函数改成等价的另一种写法）。
- 禁止为了"看起来更规范"而大规模重命名、移动文件，除非伴随实质性改进并经评估确认。
- 清理死代码 / 已注释代码是有效的（它消除歧义），但不属于"无效修改"。

**③ 不让简单逻辑变复杂**

- 一段逻辑如果能用 10 行清楚表达，就不该被拆成 3 个抽象层、2 个工具函数、1 个配置文件。
- 嵌套深度 > 3 层是危险信号——不是一定要禁止，但出现时应停顿评估是否有更平铺的表达。
- 设计模式是好东西，但"为了用模式而用模式"是本末倒置。

反例：一段 15 行的表单填值逻辑，被拆成 4 个单向调用的小函数分布在 2 个文件里，调试时要跳 3 次才能看懂数据流。

**④ 不过度防御**

只在"可能真实发生"的边界做防御，不要为"理论上不可能发生"的场景加代码。

- ✅ 用户输入 → 校验（用户可能输错）。
- ✅ 外部 API 返回值 → 校验（网络可能异常、格式可能变化）。
- ✅ 文件读取 / 路径拼接 → 校验（文件可能不存在）。
- ❌ Astro 框架保证的 `Astro.props` → 不要加 `if (!props) return null`。
- ❌ 构建期已知存在的 import JSON → 不要加 `try-catch` 兜底。
- ❌ 只在单个页面使用的脚本 → 不要加"国际化预备""多主题兼容"等未来才可能需要的分支。

**判断标准**：写防御代码前问自己——"这个异常在当前项目运行环境中，有真实路径可以触发吗？"如果答案是否，就不要写。

### 2.5 决策流：新增一个页面应走的路径

```
要新增一个页面 / URL
        │
        ▼
┌───────────────────────────┐
│ 是否需要主导航露出？       │── 否 ──┐
└──────────┬────────────────┘        │
           │ 是                       │
           ▼                          │
  改 Navbar + Footer 的导航数据       │
  （注意：两处都改，或走 §原则一       │
│   抽 data/nav.ts 单一数据源）       │
           │                          │
           ▼                          │
┌───────────────────────────┐         │
│ 是否是隐藏页（noindex）？  │         │
└──────────┬────────────────┘         │
           │                          │
     ┌─────┴─────┐                    │
     ▼           ▼                    ▼
   是隐藏页    是正常页            不进导航
     │           │                    │
     ▼           ▼                    ▼
  仍用 BaseLayout（传 noindex prop）   仍用 BaseLayout
     │           │                    │
     └─────┬─────┘────────────────────┘
           ▼
   创建 src/pages/<路由>.astro
           │
           ▼
   在页面里只写"内容编排"，套 <BaseLayout title=...>
           │
           ▼
   若有可复用 UI → 抽 Component
   若有页面专属逻辑 → 留在本页 <script>
   若有跨页逻辑 → 抽 src/scripts/
```

**核心红线**：任何页面（含隐藏页、流程内部页）都必须套 `BaseLayout`。**禁止**为了"不要导航栏"而手抄一份 `<html><head><body>`——`confirm.astro` 就是这么做的，是当前最大的屎山风险点。如果 BaseLayout 当前能力不足（如不支持隐藏导航），**扩展 BaseLayout 的 props**，而不是绕开它。

## §3 命名约定

### 3.1 文件命名

| 类型 | 规则 | 示例 |
|---|---|---|
| 页面 `src/pages/` | 全小写 **kebab-case**，嵌套路由用目录 | `index.astro`、`about.astro`、`join/confirm.astro` |
| 组件 `src/components/` | **PascalCase** | `Navbar.astro`、`WxQrcodeModal.astro` |
| 布局 `src/layouts/` | **PascalCase** | `BaseLayout.astro` |
| 工具/脚本 `src/scripts/` `src/utils/` | **kebab-case**（多词用连字符） | `pdf-guard.js`、`bilibili.ts` |
| 数据 `src/data/` | **kebab-case**，按业务域命名 | `members.json`、`articles.json`、`charter.md` |
| 静态资源 `public/assets/` | **kebab-case**，按用途分子目录 | `assets/brand/HXiao_logo.png`、`assets/qrcode/` |

### 3.2 CSS 类命名

- **业务语义类**（写入 `global.css`）：扁平 **kebab-case**，无 BEM 的 `__` / `--`。
  - ✅ `.fade-in-up`、`.card-hover`、`.nav-glass`、`.container-main`、`.print-header`
  - ❌ `.fadeInUp`（驼峰）、`.nav__link--active`（BEM）、`.btn_glow`（下划线）
- **Tailwind utility**：直接用 Tailwind 原生类，不在 `global.css` 重复定义。
- **主题色类**：通过 `tailwind.config.mjs` 映射为语义化类名——`bg-bg-main`、`text-text-primary`、`border-border`、`text-accent`。
- **类名一律英文**，禁止拼音、禁止中英混合。

### 3.3 JS / TS 命名

- 变量、函数：**camelCase** —— `fetchVideoMeta`、`shouldBlockPdfOnMobile`、`formatChineseDate`
- 接口、类型：**PascalCase** —— `VideoMeta`、`Member`、`ProjectCard`
- 常量（真正不可变的配置）：**UPPER_SNAKE_CASE** —— `WX_ID`（见 `WxQrcodeModal.astro`）
- 布尔变量/props：用 `is`/`has`/`should` 前缀 —— `isDark`、`hasNav`、`shouldBlock`

### 3.4 数据文件键名

- JSON 键：原则上 **camelCase**，与 JS 一致。
  - ⚠️ 当前 `members.json` 用了 snake_case（`core_members`、`joined_at`），是历史遗留。**新增数据文件统一 camelCase**。

## §4 代码风格约定

### 4.1 缩进与换行（强制）

- **统一 2 空格缩进**，禁止 Tab。
- 行尾 **LF**，文件末尾保留一个空行，去除行尾空格。
- 由 `.editorconfig` + `.prettierrc` 强制（见根目录配置）。

> ⚠️ 当前存在 Tab/空格混用的历史文件（`contact.astro:19-24`、`submit.astro:256-278`、`global.css` 打印区块），属于历史遗留。**新增/修改代码一律 2 空格**。

### 4.2 HTML / Astro 标签风格

- 属性值用**双引号**：`class="..."`、`href="..."`。
- 空元素自闭合 XHTML 风格：`<meta charset="UTF-8" />`、`<img ... />`、`<link ... />`。
- 属性顺序（非强制，建议）：`class` → `id` → `type/href/src` → `aria-*` → 其它。
- 长属性换行时，每行一个属性、对齐缩进（见 `Navbar.astro` 的 `<button>` 写法）。

### 4.3 注释风格（项目特有约定，请遵守）

项目已沉淀一套高度统一的注释约定：

| 场景 | 风格 | 示例 |
|---|---|---|
| `.astro` 文件首行 | `// 文件名.astro - 中文职能描述` | `// BaseLayout.astro - 全局布局组件` |
| 行内中文注释 | `// 中文说明` | `// 视差效果` |
| TS/JS 工具函数 | **JSDoc 块注释**（`/** ... */`） | 见 `pdf-guard.js`、`bilibili.ts` |
| CSS 分节 | `/* ===== 标题 ===== */` | `/* ===== 动效相关 ===== */` |

**每个 `.astro` 文件首行必须有职能注释**，无一例外。

### 4.4 中英文排版（盘古之白）

- **中文与英文/数字之间加一个空格**：`通过 BVID 从 B站 API 获取视频元数据`、`PC 端`、`GitHub Pages`。
- 标点符号：正文用中文全角（，。：；？！），代码、URL、技术标识用半角。
- 品牌名统一：`浮晓 HXiao` / `浮晓 HXiao Studio`（commit `898af55`、`75f81e4` 已统一，请勿回退）。

## §6 样式系统

### 6.1 设计 token 体系

主题色通过 **CSS 自定义属性（变量）** 定义，再由 `tailwind.config.mjs` 映射为 Tailwind 类。**禁止**在 `.astro` 里硬编码 `#AEDBFC` 这类颜色值。

```
global.css :root { --color-accent: 174 219 252; ... }
                ↓ tailwind.config.mjs 映射
                ↓
页面里用：text-accent / bg-accent / border-accent
```

完整的 token 列表见 `src/styles/global.css` 开头，涵盖：背景色、文字色、强调色、辅助色、品牌色，并为深色模式（`.dark`）定义了一整套对应值。

### 6.2 何时写 utility、何时写语义类

| 情况 | 做法 |
|---|---|
| 一次性、页面专属的布局/间距 | 直接 Tailwind utility，写在 `class="..."` |
| 跨多处复用的视觉模式（按钮样式、卡片悬停、淡入动效） | 写成语义类进 `global.css` |
| 带复杂选择器 / 伪元素 / `@media` 的样式 | 写进 `global.css`（utility 表达不了） |
| 组件专属、不想污染全局的样式 | 用 Astro **scoped `<style>`**（如 `charter.astro`、`ThirdPartyModal.astro`） |

### 6.3 打印样式规范（重要治理项）

当前打印样式（`global.css` 的 `@media print` 区块）存在两个问题：

1. **对三个打印区 ID 重复定义同一套样式**：`#print-area`、`#print-area-submit`、`#print-area-confirm` 各写一遍 `.print-header` / `h2` / `.print-meta`。
2. **漏写一个**：`.print-table` / `.print-q` / `.print-a` / `.print-confirm` 只覆盖了前两个 ID，**漏掉了 `#print-area-confirm`**——这正是"复制粘贴"必然导致的漂移。

**治理方向**：用一个公共类（如 `.print-doc`）挂到三个打印区容器上，CSS 只写一份 `.print-doc .print-header { ... }`，从根本上消除"改一处要改三处"的风险。

**新增打印页时**：必须用 `.print-doc` 公共类，禁止再复制一份针对新 ID 的样式。

### 6.4 `!important` 使用约束

- 仅允许在两类"强制覆盖"场景使用：`@media print`（强制浅色、强制隐藏非打印区）、`@media (prefers-reduced-motion: reduce)`（强制关闭动画）。
- 其它场景**禁止**用 `!important` 解决优先级问题——应通过调整选择器层级解决。

## §7 可访问性与 SEO 基线

> 以下为**每个页面都必须满足**的基线，AI 每次交付前对照自检。

### 7.1 必须项清单

- [ ] `<html lang="zh-CN">`（由 `BaseLayout` 保证，**不要**在自己写的页面里覆盖）。
- [ ] 页面有 `title`（通过 `<BaseLayout title="...">` 传入）。
- [ ] 有 `meta description`（`BaseLayout` 有默认值，重要页面应通过 prop 覆盖，参考 `projects.astro`）。
- [ ] Open Graph + Twitter Card meta（由 `BaseLayout` 注入，无需手写）。
- [ ] 所有 `<img>` 必有 `alt`，**禁止空 alt**。
- [ ] 所有图标型 `<button>` 必有 `aria-label`（如主题切换、汉堡菜单、社交链接）。
- [ ] 弹窗用 `role="dialog"` + `aria-modal="true"` + `aria-labelledby`，并支持 ESC 关闭（参考 `WxQrcodeModal.astro`、`ThirdPartyModal.astro`）。
- [ ] 动效必须响应 `@media (prefers-reduced-motion: reduce)`（`global.css` 已有全局降级，JS 动画也要判断，参考 `StarCanvas.astro`）。
- [ ] 隐藏页（如流程内部页）加 `<meta name="robots" content="noindex, nofollow">`。

### 7.2 Logo `alt` 统一（强制）

同一张 Logo 的 `alt` 当前有三种写法（`HXiao Studio` / `浮晓 HXiao Studio` / `浮晓 HXiao`），**统一为 `浮晓 HXiao Studio`**（与正文品牌标识一致）。

### 7.3 语义化标签

优先使用语义化标签：`<header>`、`<nav>`、`<main>`、`<article>`、`<section>`、`<aside>`、`<footer>`、`<time>`、`<details>`/`<summary>`（参考 `charter.astro` 的 `<article>`、`join.astro` 的 `<details>` FAQ）。

## §8 Git 协作约定

### 8.1 Commit 前缀规范（强制）

每个 commit 必须以中文类型前缀开头，格式：`[类型] 简述`。

**互斥类型集合**（收敛后）：

| 前缀 | 用于 | 示例 |
|---|---|---|
| `[新功能]` | 新增功能、新页面、新组件 | `[新功能] 集成站点地图 Sitemap 自动生成` |
| `[修复]` | 修复 bug、修复错误 | `[修复] contact 页面 GitHub 链接地址拼写错误` |
| `[优化]` | 性能、体验、可访问性、SEO 改进（不改变功能） | `[优化] 深色模式切换动画` |
| `[重构]` | 改变代码结构、不改变对外行为 | `[重构] projects 页面改造为项目聚合入口` |
| `[样式]` | 纯样式调整（CSS、Tailwind 类、视觉细节） | `[样式] 调整首页卡片间距` |
| `[内容]` | 文案、数据内容更新（不改代码逻辑） | `[内容] 更新成员列表` |
| `[配置]` | 配置文件、依赖版本、构建配置 | `[配置] 升级 Astro 至 6.4.8` |
| `[部署]` | CI、部署、域名相关 | `[部署] 配置 GitHub Actions 自动发布` |
| `[文档]` | 文档变更（含本规范文档） | `[文档] 新增开发规范` |

**已废弃的前缀**（不要再使用）：
- ~~`[调整]`~~ → 用 `[内容]` 或 `[优化]`
- ~~`[整理]`~~ → 用 `[重构]` 或 `[优化]`
- ~~`[内容调整]`~~ → 用 `[内容]`

**质量要求**：
- 类型与内容匹配，不要用 `[新功能]` 包装一个纯 bug 修复。
- 简述用祈使句或陈述句，简洁，中英文间加空格。
- 历史上出现过漏写右括号 `]` 的情况（`d9d74c1 [内容调整 补充视频数据`），提交前检查括号闭合。

### 8.2 提交粒度

- **一个 commit 做一件事**：不要把"加新功能 + 改无关 bug + 改样式"塞进一个 commit。
- 单人项目可以稍宽松，但遇到重构时，**每个治理点独立一个 commit**（便于回滚和 review）。

### 8.3 分支与发布

- `main` 分支即生产分支，push `main` 自动触发部署。
- 较大的改动建议开分支开发，合并到 `main` 再部署。

## §9 AI Coding 决策树与 Do/Don't 清单

> 这一章是给 AI（和快速自查的人类）的"可执行核心"。开工前看决策树，交付前对照 Do/Don't。

### 9.1 开工前决策树

#### A. 新增 / 修改页面

```
这个页面是否套了 <BaseLayout>？
├── 否 → ❌ 错误。必须套 BaseLayout。
│        若 BaseLayout 能力不足（如要隐藏导航/设 noindex），扩展 BaseLayout 的 props，
│        绝不绕过它手写 <html><head><body>。
└── 是 → ✅ 继续
        页面里的 UI 是否会被 ≥2 个页面用到？
        ├── 是 → 抽成 Component
        └── 否 → 留在本页
```

#### B. 新增 / 修改组件

```
这个组件的数据来源？
├── 硬编码业务数据（邮箱、链接、ID） → ❌ 改为从 src/config.ts 或 src/data/ 引入
├── 父级传入（props） → ✅ 正确
└── 组件内部 import 业务页面 → ❌ 错误。组件不能反向依赖页面
```

#### C. 写一段 JS 逻辑

```
这段逻辑是否已经在别处出现过？
├── 是，这是第 2 次 → 复制可以，但在原处留 // TODO: 与 xxx 重复，考虑抽象
├── 是，这是第 3 次及以上 → ❌ 必须抽到 src/scripts/ 公共函数，禁止再复制
└── 否，第 1 次 → 直接写
```

#### D. 写样式

```
这个颜色值是 hex/rgb 吗（如 #AEDBFC）？
├── 是 → ❌ 改用设计 token（text-accent / bg-bg-main 等）
└── 否 → ✅
这段样式是否针对"多个相似容器"重复定义（如 #print-area、#print-area-submit、#print-area-confirm）？
├── 是 → ❌ 改用公共类（.print-doc），只写一份
└── 否 → ✅
```

#### E. 加内容数据

```
这份内容会持续增删吗（如成员列表、动态）？
├── 是 → 进 src/data/*.json，页面用 .map() 渲染
└── 否，一次性静态文案 → 写在 .astro 里
这个常量（邮箱/链接/ID）会被 ≥2 处引用吗？
├── 是 → ❌ 禁止硬编码，进 src/config.ts
└── 否 → 可以写死
```

### 9.2 Do / Don't 总表

#### ✅ Do

1. **每个页面套 `BaseLayout`**，通过 props 传 `title` / `description`。
2. **每个 `.astro` 文件首行写职能注释**：`// 文件名.astro - 中文描述`。
3. **站点常量集中**到 `src/config.ts`，禁止散落硬编码。
4. **导航数据单一来源**（待抽取后生效）。
5. **颜色用设计 token**（`text-accent` 等），不用 hex 值。
6. **第 3 次重复的逻辑必须抽象**成组件 / 工具函数 / 数据源。
7. **`<img>` 必有 `alt`**，Logo 统一 `alt="浮晓 HXiao Studio"`。
8. **图标 `<button>` 必有 `aria-label`**。
9. **弹窗用 `role="dialog"` + `aria-modal` + ESC 关闭**。
10. **动效响应 `prefers-reduced-motion`**。
11. **缩进 2 空格**，由 `.editorconfig` / `.prettierrc` 强制。
12. **中英文间加空格**（盘古之白）。
13. **commit 用 `[类型] 简述` 前缀**，类型用 §8.1 的互斥集合。
14. **文件命名遵守 §3.1**：页面 kebab-case、组件 PascalCase。
15. **新增打印页用 `.print-doc` 公共类**（待打印样式重构后生效）。
16. **开发只在 `src/` 内进行**，`dist/` 勿手改、`public/` 只放原样资源。
17. **数据驱动优先**：会增删的内容走 `src/data/*.json`。
18. **一个 commit 做一件事**，重构时每个治理点独立提交。
19. **CSS `!important` 只用于** `@media print` 和 `prefers-reduced-motion` 两类强制覆盖。
20. **TS 用 strict 模式**，工具函数配 JSDoc。
21. **优先平铺直叙**，复杂逻辑拆成可读步骤而非多层抽象。
22. **防御代码只加在有真实触发路径的边界**（用户输入、外部 API、文件 I/O）。

#### ❌ Don't

1. **不要绕过 `BaseLayout` 手写 `<html><head><body>`**（即使是隐藏页——`confirm.astro` 是反面教材）。
2. **不要在 `.astro` 里硬编码邮箱/链接/ID 等站点常量**。
3. **不要散落重复逻辑**——同一段脚本第 3 次出现必须抽象。
4. **不要复制粘贴 CSS 选择器**对应多个 ID——用公共类。
5. **不要用 hex/rgb 颜色值**，用设计 token。
6. **不要用拼音或中英混合的类名/变量名**。
7. **不要让组件反向 import 页面**（违反分层依赖方向）。
8. **不要在叶子层（Data/Style/Script）引用上层（Component/Page）**。
9. **不要用 `!important` 解决优先级**（除两类强制覆盖场景）。
10. **不要用已废弃的 commit 前缀**（`[调整]`/`[整理]`/`[内容调整]`）。
11. **不要手动改 `dist/`**，那是构建产物。
12. **不要把"加功能 + 修 bug + 改样式"塞进一个 commit**。
13. **不要给 `<img>` 留空 `alt`**。
14. **不要新增针对单个打印区 ID 的样式副本**——统一走 `.print-doc`。
15. **不要绕过 `main` 分支部署**。
16. **不要在第 1 次使用时提前抽象**（YAGNI：等真正出现第 2、3 次复用再抽）。
17. **不要做不改变行为、仅为"换种写法"的修改**（纯重命名、等价改写）。
18. **不要为"理论上不可能触发"的场景加防御代码**（如 try-catch 兜底已知存在的 import JSON）。
19. **不要让嵌套深度 > 3 层**而不评估平铺方案。

## §10 工程化工具链

本项目已落地以下可执行约束：

| 工具 | 文件 | 作用 |
|---|---|---|
| **EditorConfig** | `.editorconfig` | 统一缩进（2 空格）、换行（LF）、字符集（UTF-8）、去行尾空格。编辑器原生支持，无需装包。**对 `.astro` 文件同样生效**，是当前 `.astro` 格式的主要约束。 |
| **Prettier** | `.prettierrc` + `.prettierignore` | 格式化约束。已装 `prettier-plugin-astro` 备用，但**当前 `.astro` 文件被 `.prettierignore` 暂时排除**（原因见下注），Prettier 实际只格式化 `.ts/.js/.css/.json/.mjs` 等非 Astro 文件。 |
| **npm scripts** | `package.json` | `npm run format`（一键格式化）、`npm run format:check`（检查不改动）。 |

> ⚠️ **Astro 6 格式化空窗说明**：`prettier-plugin-astro` 当前最新版（0.14.1）只支持到 Astro 5，遇到本项目 `.map()` 表达式内的 HTML 注释会解析崩溃。这是 Astro 6 太新导致的生态空窗，非本项目问题。临时方案：`.prettierignore` 排除 `*.astro`，由 EditorConfig 兜底基础格式。**待插件发布 Astro 6 适配版后，删除 `.prettierignore` 里的 `*.astro` 一行即可全量启用**——届时也建议同步更新本节。

**使用方式**：
- 写完代码执行 `npm run format` 自动格式化。
- 想检查代码是否符合规范（不改文件）执行 `npm run format:check`。
- 大批量历史文件的格式化（如缩进修复）建议单独一个 `[重构]` commit。

**未引入（按需再考虑）**：
- ESLint（JS/TS 逻辑检查）——当前规模未必要，规模增长或多人协作时再引入。
- commit-msg 钩子（校验 `[类型]` 前缀）——单人项目靠自觉即可，多人协作时引入 husky + commitlint。
- `prettier-plugin-tailwindcss`（自动排序 Tailwind 类）——会改动所有现有文件引入 diff 噪音，留作后续可选。
